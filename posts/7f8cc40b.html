并行思考

执行的抽象/机制

硬件实现并行

思考效率：fast != efficient

现代多核处理器

现代处理器中的并行执行形式：

  • 多核：使用多线程 pthread
  • SIMD：如 AVX
  • 超标量

内存延迟：

  • cache
  • prefetch
  • 多线程隐藏

GPU

并行编程抽象

ISPC：Intel SPMD Program Compiler

ISPC 关键词：

  • uniform：所有实例的变量都有相同的值
  • programCount：实例数
  • programIndex：当前实例的 id

分解方案通常是交错执行

两者各有优劣，一般来说对于数据无法预测的情况下使用交错执行会比较好

ISPC 的 gang 抽象是 SPMD，但是实现是 SIMD

但其还有另一个抽象 task，用于实现多核执行

三个模型：

共享地址空间模型，通信方法为：

  • 读写共享的变量
  • 同步
  • 单处理器编程逻辑的延伸

对于内存不在一起的情况，需要总线连接

NUMA：非一致性内存访问

消息传递模型：通过收发消息进行通信

数据并行模型：基本思想是 map 函数到大量数据集上，stream 编程模型，使用 gather 和 scatter 进行通信

这几种模型经常混合使用

并行编程基础

![创建并行程序](/illustration/CMU 15-481：并行计算架构/创建并行程序.webp)

分解时最重要的是确定任务之间的依赖关系

静态分配 vs 动态分配

协调的目标是减少通信/同步的损失

对于网格，可以通过红色和黑色交替的顺序更新以减少依赖，块状划分比较好

如果通过共享地址空间，则需要使用 lock 来保护总和变量

如果使用消息传递，两个块之间行的通信可以使用 ghost cell

工作分布和调度

关键目标：

  • 平衡负载
  • 减少通信
  • 减少额外工作

选择任务大小：

  • 较小的任务可以减少负载不平衡
  • 较大的任务可以减少管理分配的开销

对于动态分配，可以每个工作线程都有一个任务队列，如果某个队列为空则从其他线程偷取任务

cilk 是一个并行编程的库，可以自动调度任务

  • cilk_spawn foo(args) 创建一个新的任务
  • cilk_sync 等待所有任务完成

其中 cilk_spawn 会创建一个并行的新任务，但是不一定会立即执行

两种方案：先继续执行当前任务，再执行新任务；或者立即执行新任务。

在 cilk 中，会选择运行 spawned 的任务

在队列偷取时，可以选择随机偷取，也可以选择最后一个任务

GPU 架构和 CUDA 编程

运行在 CPU 上的部分：（host）

CUDA 核定义：（device）

CUDA 内存模型：

  • 全局内存：所有线程都可以访问
  • 共享内存：一个 block 内的线程可以访问
  • 寄存器：每个线程都有自己的寄存器

局部性、通信和竞争

一个非流水线通信的简单模型：

T(n)=T0+nBT(n) = T_0 + \frac{n}{B} T(n)=T0​+Bn​

其中

  • T0T_0T0​：通信的固定开销
  • BBB：带宽
  • nnn：数据量
  • T(n)T(n)T(n)：通信时间

更广泛地说，总通信时间 = 开销 + 占用 + 网络时延，而总通信开销 = 通信时间 - 重叠

计算强度 = 计算数量 / 通信数量

例如对于计算洋流，可以划分为一个个正方形网格来减少通信，计算强度为 NP\frac{N}{\sqrt{P}}P​N​

第四种 cache miss：通信 miss

可以通过 fusing loop 来提高时间局部性

在方格的边界上，空间局部性非常差，一种方法重排块中的数据布局，使得一个块中的数据在内存中是连续的

减少竞争的方法：

  • 分布式工作队列
  • 粒子格的数据结构

并行将粒子划分到所在的格子中的一种方法：数据并行

  • 单独计算每个粒子所在的格子
  • 根据格子并行排序
  • 实际上已经划分完成了，可以为某个各自赋予开始和结束的索引指针

Case Study

Barnes-Hut 算法：用于计算 N 个粒子的引力

  • 建立树结构
  • 计算内部结点
  • 对于每个粒子，遍历树来累加力，然后更新位置

由于每个节点处理的粒子数量不同，可以使用 cost zone 来分配任务

另一种划分方法——ORB Orthogonal Recursive Bisection：将空间划分为工作相同的子空间

scan 算法：

  • scan_inclusive：[a0,a0+a1,a0+a1+a2,⋯ ][a_0, a_0 + a_1, a_0 + a_1 + a_2, \cdots][a0​,a0​+a1​,a0​+a1​+a2​,⋯]
  • scan_exclusive：[I,a0,a0+a1,a0+a1+a2,⋯ ][I, a_0, a_0 + a_1, a_0 + a_1 + a_2, \cdots][I,a0​,a0​+a1​,a0​+a1​+a2​,⋯]

对于一个数组，计算每个元素之前的和

scale 约束：

  • 面向应用：
    • 每个处理器的粒子数
    • 每个处理器的事务
  • 面向资源：
    • 问题约束
    • 内存约束
    • 时间约束

一般来说，可以将问题的尺寸视为参数的组合

trace-driven simulation：通过记录每个处理器的活动来模拟并行程序

execution-driven simulation：通过模拟每个处理器的活动来模拟并行程序

pareto curve：在资源约束下，问题规模和性能的关系

roofline model：上升的部分通常是由于内存带宽受限，水平的部分通常是由于计算受限

![roofline](/illustration/CMU 15-481：并行计算架构/roofline.webp)

建立 watermark 的方法：

  • 添加计算
  • 移除计算，加载数据
  • 改变所有对 A[0] 的数组访问
  • 移除所有原子操作或锁

这样有利于我们寻找性能瓶颈

基于 Snoop 的缓存一致性

Snoop：在 cache 中查找数据时，同时查找其他 cache 中是否有相同的数据

MESI：缓存对象有 Modified、Exclusive、Shared、Invalid 四种状态

![MESI](/illustration/CMU 15-481：并行计算架构/MESI.webp)

inclusion property：如果一个 cache 中有数据，那么所有上级 cache 中也有，这需要另外维护

基于目录的缓存一致性

使用目录来实现可伸缩的缓存一致性：通过将关于行的状态的信息存储到一个地方（目录）来避免广播

![目录read](/illustration/CMU 15-481：并行计算架构/目录 read.webp)

![目录write](/illustration/CMU 15-481：并行计算架构/目录 write.webp)

减少目录存储的开销：

  • 减少缓存行大小，将多个处理器分到一个目录节点中
  • 限制指针
  • 稀疏目录
