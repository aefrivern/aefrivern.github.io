Go风格指南

发表于2024-11-28|更新于2024-12-19|工具
|总字数:2014|阅读时长:6分钟

基于 Google Go Style 编写

命名

命名：

  • 包名全小写
  • 测试、基准测试文件名以 _test 结尾

接受者变量一般只有 1-2 个字符，通常是该类型的缩写，并保持一致：

均使用驼峰命名法，导出符号首字母大写，因此对于导出的常量写法是这样的：

一些本身包含大小写的专有名词，如

  • XML：
    • 导出：XML
    • 非导出：xml
  • iOS：
    • 导出：IOS
    • 非导出：iOS

不使用 get 前缀，直接调用函数名即可。如果是需要计算等消耗时间的获取，使用 Compute和 Fetch 前缀

变量名：作用域越大，命名越详细，但注意不要重复该作用域

在命名时，不要重复该符号的

  • 上下文

  • 类型

  • 参数和返回值

    • 例外：需要通过参数或返回值来区分，如

    • func ParseInt(input string) (int, error)
func ParseInt64(input string) (int64, error)

包可以提供一个 doc.go 文件，用于描述包的整体功能

尽量提供一些可运行的样例，该样例应该放在 test 文件中，而不是在源代码文件

  • 如果无法提供可运行的样例，可以在代码注释中提供一个简短的代码片段

给返回值命名可以更好地描述该返回值的含义和用途

如果一个包分了多个文件，注意只有一个文件可以包含 package 注释

函数注释：

  • 并不是所有的参数（包括函数参数、结构体域、可选 API）都需要在注释中进行描述，只有那些不明显的参数和容易使用出错的参数才需要描述
  • 如果是一个明显只读的函数，不需要注明是并发安全的，但是可变操作需要注明
  • 如果需要客户端做一些额外的清理工作，必须在注释中明确说明，必要时给出示例步骤

错误：

  • 函数注释中写出函数返回的错误的类型
  • 在包文档中列出可能返回的错误

Godoc：

  • 使用空行分割不同的段落
  • 使用 // 注释，而不是 /* */ 注释
  • 测试文件可以包含可运行的示例关联到 godoc 中的文档
    • 使用 // Output: 注释来标记示例的输出
    • 使用 // Unordered: 注释来标记无序的输出
    • 使用 // Error: 注释来标记示例的错误输出
  • 使用两个缩进来标记代码块

通常来说，约定大于注释，例如

import

只有为了符合包名的命名规范或发生名称冲突时，才会在导入时使用别名

尽量避免使用 . 导入

语言

外部包的类型必须指出域名

避免重复的域名

零值域一般可以忽略

nil 比较糟糕，尽可能选择零值或显式返回错误

变量声明初始化：

  • 非零值：i := 42
  • 零值：var primes []int

测试

如果某函数的实际输出和预期输出一致，可以使用表驱动测试 来测试相似的逻辑：

但如果行值决定了测试内部的不同行为，则最好分开来提高可读性

可以使用 T.helpers() 来标记辅助函数，这样可以在测试失败时显示辅助函数的调用栈

测试一般定义在不同包，名称为被测试包名后加 _test 后缀

一般来说，应该将逻辑（验证和失败）放在 Test 函数中

  • 如果无法简单地验证，则可以定义函数，该函数应该返回一个 error 而不是接受 testing.T 作为参数

测试失败时，打印的应该包括三部分，格式类似于：YourFunc(%v) = %v, want %v

对于一个整体（数组，结构体等），直接将其作为一个整体进行比较

尽可能一次性打印出所有的问题，所以推荐使用会继续执行的 t.Error 而不是 t.Fatal

cmp 包是一个专门为测试设计的包，可以用来比较两个值

  • 如果参与比较的两个值比较大，可以使用 cmp.Diff 来打印出两个值的差异

字符串连接

一般情况推荐 +

需要格式化时，推荐使用 fmt.Sprintf

如果要连接多个字符串，推荐使用 strings.Builder

项目结构

来自 project-layout

  • cmd：应用程序的入口点
  • internal：私有应用程序和库代码
  • pkg：公共库代码
  • api：OpenAPI/Swagger 规范， JSON schema 文件，协议定义文件
  • web：Web 应用程序特定的组件：静态 Web 资源，服务器端模板和 SPAs
  • configs：配置文件模板或默认配置
  • scripts：脚本文件，使得根目录下的 Makefile 更简洁
  • build：构建和持续集成
  • deployments：IaaS，PaaS，系统和容器编排部署配置和模板
  • test：额外的测试应用和测试数据

其他目录：

  • docs：设计和用户文档
  • tools：支持项目的工具
  • examples：示例应用和/或使用库的示例
  • third_party：外部帮助工具，分叉代码和其他第三方工具（例如 Swagger UI）
  • assets：与项目相关的其他资产
  • website：项目的文档和站点

注意不要有 src 目录
