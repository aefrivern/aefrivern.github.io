UCB CS161：计算机系统安全

发表于2024-11-20|更新于2024-11-26|学习笔记
|总字数:5165|阅读时长:16分钟

安全原则

威胁模型 threat model：一个关于攻击者和其拥有的资源的模型（不需要防范任何人，只需要防范攻击者）

可信计算基 trusted computing base：安全所依赖的系统组件，一般来说越小越好（KISS 原则）

考虑人类因素：人类是安全的薄弱环节，不要过度依赖人类的行为，尽量做成傻瓜式

安全是经济：防御的预期收益应该攻击的预期花费成正比

如果无法预防，就检测：

  • 威慑 deterrence：在攻击发生前停下
  • 预防 prevention：在攻击发生时停下
  • 检测 detection：知道攻击发生了
  • 回复 response：在攻击发生后做一些事

靠深度防御：多种类型的防御可以分层

最少权限原则：只给予程序运行所需的权限

分离职责：需要不同的部分才能运行

确保完全仲裁：每个接入点都被监视和保护

香农准则 Shannon’s Maxim：敌人知道系统的设计（不能指望隐藏的安全性）

使用失败后安全：系统应该在失败后保持安全

从开始时就考虑安全：安全应该是系统设计的一部分

密码学简介

三个目标：保密性，完整性，认证

基本策略单元：密钥

两种钥匙模型：对称密码模型（加密和解密使用相同的密钥），非对称密码模型（加密和解密使用不同的密钥）

Kerckhoffs 原则：系统的安全性不应该依赖于系统的设计细节，只依赖于密钥的保密性

在这门课程中，选择的威胁模型是选择明文攻击模型，即可以让别人加密自己给出的明文

对 保密性 的定义：密文不会泄露明文的附加信息

一个测试该定义的实验游戏：

  • Eve 选择两个相同长度的信息
  • Alice 选择一个信息加密并发送给 Bob
  • Eve 读了密文后猜测 Alice 选择的信息
  • 如果 Eve 猜对的概率为 1/2，那么这个加密算法就是保密的

IND-CPA：在选择明文攻击模型下，一个加密算法是 IND-CPA 安全的，当且仅当 Eve 猜测的概率为 1/2

在实际情况下，一般会受限于运行时间，此外，如果猜对的概率稍大于 1/2，也是可以接受的

块密码

One-Time pad 可用，two-time pad 就不行，但问题在于消耗了大量密钥，且需要提前通过安全信道传递密钥

块密码 Block Cipher：将明文分成固定长度的块，对每个块进行加密，特点在于密钥的长度和块的长度可以不同

AES Advanced Encryption Standard：对称密码，块长度为 128 位，密钥长度为 128/192/256 位，是一种标准块密码

当然，由于这种方案是 确定性 的，因此不是 IND-CPA 安全的，因此需要引入 操作模式

将明文分成块，对每个块使用相同的密钥加密，但是这种模式是不安全的，因为相同的明文会加密成相同的密文，因此这里使用 Initialization Vector IV，即在每个块加密时引入一个随机的 IV，这样即使相同的明文也会加密成不同的密文

为了减少 IV 的数量，可以只在第一个块使用 IV，后续块使用前一个块的密文作为 IV，这种模式称为 CBC 模式

看这张过程图，可以发现加密过程无法并行，但是解密过程可以并行

对于不是块大小的倍数的明文，可以使用 填充 的方式，即在明文末尾加入一些无意义的数据，解密时再去掉这些数据

  • PKCS#7：每个填充字节的值等于填充的字节数

如果重用了 IV，那么会泄露从相同明文块到第一个不同明文块的信息

CTR 模式：将 IV 作为计数器，每个块加密时计数器加一，通过 EBC 生成密钥，然后使用 One-Time pad 加密，这样可以并行加密和解密

这种模式也不需要填充，因为可以直接丢弃多余的密钥

两种模式的比较：

  • CBC 性能差一点，但是如果出了问题也只会泄露部分信息
  • CTR 性能强，但是如果使用不当（nouce 重用），会引发灾难性后果

伪随机数生成和 Diffie-Hellman 密钥交换

一般来说，获取到的物理熵源都是有偏的，通常需要将多个熵源混合在一起得到一个相对不错的熵

伪随机数生成 PRNG 的函数：

  • Seed：输入一个种子
  • Reseed：使用现有的状态和熵更新内部状态
  • Generate：生成一个随机数

PRNG 的安全性：

  • 对于 s 位的初始化种子，只有 2s2^s2s 个可能的种子，但是 PRNG 的状态空间大于 2^s，故无法做到真正随机
  • 之前说了，只要超过的范围在可接受范围内，那么就在计算上不可区分
  • 简单来说，这种性质是：攻击者无法预期 PRNG 未来的输出

PRNG 回滚抵抗性：

  • 如果 PRNG 的状态泄露，那么之前的输出也是安全的
  • 尽管不是必要的，但这种性质是 PRNG 的一个很好的特性

HMAC-DRBG：

  • 生成方法：循环调用 HMAC，每次输入上一次的输出和一些熵，然后输出一些随机数

应用：UUID

流密码 stream cipher：将 PRNG 的输出作为 One-Time pad 的密钥，之前提到的 AES-CTR 就是一种流密码

一个形象化地解释 Diffie-Hellman 密钥交换的方法：

其中生成器要求满足 discrete logarithm problem 离散对数问题：给定 ggg，ppp，gxmod  pg^x \mod pgxmodp，求 xxx 非常困难

Diffie-Hellman 假设：给定 ggg，ppp，gxmod  pg^x \mod pgxmodp，gymod  pg^y \mod pgymodp，求 gxymod  pg^{xy} \mod pgxymodp 非常困难

Diffie-Hellman 可以暂时地使用（DHE Diffie-Hellman Ephemeral），即在获得 K=gxymod  pK = g^{xy} \mod pK=gxymodp 后，立即销毁 xxx 和 yyy，这样即使攻击者获得了 KKK，也无法获得 xxx 和 yyy

  • 其中的 KKK 被称为 session key 会话密钥
  • 优点在于能够前向保密，即即使攻击者获得了 KKK，也无法获得之前的通信内容

但是这种方法会受到中间人攻击，即两人实际上在和中间人通信，因此 DHE 是一种 主动协议，即双方必须同时在线来交换密钥，因为 Diffie-Hellman 无法提供认证

Elliptic Curve Cryptography ECC：一种基于椭圆曲线的密码学，其离散对数问题更难，因此 ECC 的密钥长度可以更短，但是 ECC 的实现更复杂

公钥加密和数字签名

分为三步：

  • 生成公钥和私钥
  • 公钥加密
  • 私钥解密

ElGamal 加密：

  • 生成方法：选择一个大素数 ppp，一个生成器 ggg，一个私钥 xxx，一个公钥 y=gxmod  py = g^x \mod py=gxmodp
  • 加密方法：选择一个随机数 rrr，计算 c1=grmod  pc_1 = g^r \mod pc1​=grmodp，c2=m⋅yrmod  pc_2 = m \cdot y^r \mod pc2​=m⋅yrmodp
  • 解密方法：m=c2⋅(c1x)−1mod  pm = c_2 \cdot (c_1^x)^{-1} \mod pm=c2​⋅(c1x​)−1modp

但是这个方法不是很好，不是 IND-CPA 安全的，而且会被篡改

RSA 加密：

  • 生成方法：选择两个大素数 ppp，qqq，计算 n=p⋅qn = p \cdot qn=p⋅q，ϕ(n)=(p−1)⋅(q−1)\phi(n) = (p-1) \cdot (q-1)ϕ(n)=(p−1)⋅(q−1)，选择一个 eee，使得 gcd(e,ϕ(n))=1gcd(e, \phi(n)) = 1gcd(e,ϕ(n))=1，计算 d=e−1mod  ϕ(n)d = e^{-1} \mod \phi(n)d=e−1modϕ(n)
  • 加密方法：c=memod  nc = m^e \mod nc=memodn
  • 解密方法：m=cdmod  nm = c^d \mod nm=cdmodn
  • 其中的 nnn 和 eee 为公钥，ddd 为私钥

RSA 问题：给定 nnn，eee 和 c=memod  nc=m^e \mod nc=memodn，求 mmm 非常困难

RSA 同样不是 IND-CPA 安全的，因此需要引入 填充，一种常见的填充方法是 OAEP，即将明文加密后再加密

数字签名：使用私钥对消息进行签名，然后使用公钥进行验证

证书，密码哈希

trust-on-first-use(TOFU)：第一次使用时信任，之后不再信任。因为如果不相信任何东西，那么就无法建立信任，都需要一个信任锚点。

证书 certificate：一个公钥和一些元数据的集合，由一个 certificate authority CA 签名，证明了公钥的真实性

撤销证书的方法：

  • 每个证书都有一个有效期
  • 周期性发布无效的证书列表

密码哈希：将一个任意长度的消息映射到一个固定长度的哈希值，常见的密码哈希函数有 SHA-256，SHA-3，MD5

应对提前计算的攻击：

  • salt：在哈希值中加入一些随机的数据，这个 salt 不是保密的（类似于 IV 和 nouce）
  • 使用更慢的哈希函数

在线攻击的防御：超时

交通分析：分析何时在和谁通信，即获取了一些元数据

边信道攻击 side channel：方案本身不存在问题，但在实现中可以会泄露一些明文的信息

在密码学中使用到了许多常数，尽管从理论上来说，这些常数选什么都行，但是一些特定的数值选择可能会相比别的数值更容易泄露信息，而制定这个标准的人会从这里给自己留下后门（所以才要有各种国产的加密算法，毕竟美国也不是没有干过这种事）

而制定这个标准的人为了应对别人的质疑，需要给出一些证据证明这个数是随意选取的

Web

Web 中的风险：

  • 服务器应该预防未授权的访问——服务器端的安全
  • 恶意网站不能攻击电脑——沙盒
  • 恶意网站不能篡改信息或与其他网站交互——同源策略

Cookie 和 CSRF

Cookie：服务器发送给浏览器的一些信息，浏览器会保存在本地，之后每次请求都会带上这些信息

  • name 和 value：键值对
  • domain：可以访问这个 cookie 的域名
  • path：可以访问这个 cookie 的路径
  • secure：只有在 HTTPS 下才会发送
  • httpOnly：只能通过 HTTP 访问，不能通过 JavaScript 访问
  • expires：过期时间

域名属性是服务器域名的一个后缀，但不能是顶级域名；路径属性是 URL 的一个前缀

会话 session：一系列相同用户的请求和相应

session token：服务器生成的一个随机数，用来标识一个会话，发送给客户端。在客户端发送请求时，会带上这个 token，服务器通过这个 token 来识别用户

CSRF Cross-Site Request Forgery：攻击者通过伪造请求，让用户在登录状态下执行一些操作

策略：

  • GET：欺骗用户点击链接，或是在网站上放 HTML

  • POST：网站上的某些 JavaScript

防御方法

  • CSRF Token：在请求头中加上一个攻击者不知道的秘密值
  • referer header：由于 CSRF 攻击中，受害者总是从不同的网站发送恶意请求，所以可以在 HTTP 的请求头中指示请求的来源
  • SameSite cookie：在 Cookie 中加入一个 SameSite 属性，使得改 cookie 只能在同源请求中发送

XSS 和 UI 攻击

XSS Cross-Site Scripting：攻击者在网站上注入一些恶意的脚本，当用户访问这个网站时，这些脚本会被执行

  • Stored XSS：攻击者将恶意脚本存储在服务器上，当用户访问这个网站时，这些脚本会被执行
  • Reflected XSS：攻击者将恶意脚本放在 URL 中，当用户访问这个 URL 时，这些脚本会被执行

防御方法：

  • HTML 消毒：将其中的特殊字符转义
  • CSP Content Security Policy：限制网站中可以执行的脚本

UI 攻击：攻击者通过伪造网站的 UI，让用户误以为是真的

  • clickjacking：在网站上放置一个透明的 iframe，用户点击的时候实际上点击的是这个 iframe 上的按钮
  • phishing：伪造网站的 UI，让用户输入一些敏感信息

两因素认证 2FA：在输入密码后，还需要输入一个验证码

SQL 注入和 CAPTCHA

SQL 注入：攻击者通过在输入框中输入一些 SQL 语句，来执行一些恶意操作

blind SQL injection：攻击者无法直接看到数据库的返回结果，只能通过一些结果来判断是否注入成功

防御方法：

  • 输入消毒
  • 使用预编译语句

问题不光出现在 SQL 中，例如 命令注入 command injection，攻击者通过在输入框中输入一些命令，来执行一些恶意操作

CAPTCHA Completely Automated Public Turing test to tell Computers and Humans Apart：一种区分人类和机器的测试

问题在于随着 AI 的发展，这种测试变得越来越容易被破解，同时也将一些人类排除在外

低级网络攻击

ARP spoofing：攻击者发送一个 ARP 包，将受害者的 IP 地址映射到攻击者的 MAC 地址

防御方法：使用 switch

WPA 2 Wi-Fi Protected Access 2：一种 Wi-Fi 加密协议，使用 4-way handshake 来建立连接：

其中：

  • ANonce：AP 生成的随机数
  • SNonce：客户端生成的随机数
  • PSK Pre-Shared Key：预共享密钥，即 Wi-Fi 密码
  • PTK Pairwise Transient Key：会话密钥
  • MIC Message Integrity Code：消息完整性码
  • GTK Group Transient Key：组密钥，用于多播

注意到有些数据可以合并起来一起发送：

WPA-PSK 的攻击手段：

  • 骗子 AP
  • 离线暴力攻击
  • 非前向安全

WPA-Enterprise：使用由服务器随机生成的密钥

IP 地址欺骗 IP spoofing：攻击者发送一个带有伪造 IP 地址的数据包

BGP hijacking：攻击者通过伪造路由信息，将流量重定向到自己的服务器

TLS

TLS Transport Layer Security：一种安全传输协议，用于保护网络通信的安全

TLS 的握手过程：

  • Client Hello：客户端发送一个包含自己支持的加密算法的列表
  • Server Hello：服务器选择一个加密算法，发送一个证书
  • 使用 RSA 或 DH 交换密钥，双方最终得到四把密钥，用于加密和验证
  • 验证之前的通信信息没有被篡改

重放攻击：攻击者通过重复发送相同的数据包来进行攻击

防御方法：

  • 重放之前 TLS 连接的信息：每次握手使用不同的 RBR_BRB​ 和 RSR_SRS​，这样每次握手生成的对称密钥都不同
  • 重放当前 TLS 连接的信息：每个消息都有一个唯一的序列号（注意不是 TCP 序列号）

前向安全：RSA 实现方案不保证前向安全，DHE 实现方案保证前向安全，因此现在一般都使用 DHE

TLS 提供了 端到端的安全

DNS

Bailiwicked checking：解析器只接受处于相同服务器域的记录

on-path 攻击者可以向解析器中插入恶意的记录

防御方法：DNS 请求中有一个随机的 ID number，攻击者返回的响应也必须包括这个 ID number，比较难猜对

此外，由于缓存有一定的有效期，故当某次抢占失败后，必须要等待一段较长时间才可以再次尝试，但 Kaminsky 攻击 可以跳过这个等待期

  • 依次请求 fake1.google.com 等，响应中附加上 www.google.com 的假地址，求解器会将这个假地址缓存下来
  • 防御方法：不要缓存 glue records

DNSSEC

我们需要响应的完整性而不是保密性

尽管 TLS 提供了端到端的安全，但不能保证 DNS 全过程的安全

DNSSEC 是向后兼容的

PKI：阻止恶意域名服务器

拒绝服务和防火墙

应用级 DoS 防御：识别、孤立、限额、工作流证明、过度配置资源

网络级 DoS 防御：包过滤、过度配置资源

SYN 泛洪：攻击者发送大量 SYN 包，但不发送 ACK 包，导致服务器资源耗尽

  • 防御方法：SYN cookies，服务器不保存 SYN 请求，而是将 SYN 请求的信息加密后发送给客户端，客户端发送 ACK 包时，服务器解密后验证

网络访问由一个 政策 控制

  • 默认允许政策
  • 默认拒绝政策

防火墙一般是一个包过滤器

  • 无状态
  • 有状态

问题在于防火墙看到的和服务器看到的是不一致的，例如 TTL 不同，所以可以控制某些东西防火墙能看到，而后面的主机看不到

此外，防火墙的另一个问题在于难以防范内部的攻击

入侵检测

网络入侵检测系统 NIDS：在网络上安装的检测器，在局域网和互联网之间

同样，缺点在于该检测器和用户看到的可能会不一致，尤其是加密的流量

主机入侵检测系统 HIDS：在主机上安装的检测器

问题是开销较大，需要在每台主机上安装

日志 logging：记录系统的活动，可以在后面分析

问题在于攻击检测不是实时的

基于签名的检测 signature-based detection：即 黑名单 blacklisting

基于规范的检测 specification-based detection：即 白名单 whitelisting

基于异常的检测 anomaly-based detection：训练一个正常表现的模型而不是手动指定正常表现

行为检测 behavioral detection：寻找危害的证据，和签名的不同，不关注输入，而是关注系统执行的行为

脆弱性扫描 vulnerability scanning：先攻击自己的系统，然后对成功的攻击进行防御

蜜罐 honeypot：一个虚拟的系统，用于吸引攻击者

取证 forensics：在系统被攻击后，分析攻击的痕迹

入侵预防系统 IPS：在检测到攻击后，自动阻止攻击

恶意软件和硬件问题

恶意软件 Malware：是在受害者电脑上运行的攻击者的代码

  • 病毒 virus：需要一个宿主程序，通过复制自己来传播
  • 蠕虫 worm：自己传播，不需要宿主程序

传播策略：

  • 注入到其他程序中
  • 通过网络传播

检测策略：特征检测 signature-based detection：检测已知的恶意软件

攻击者与杀毒软件之间的博弈，其中攻击者有一点优势

多态代码 polymorphic code：每次传播时都会插入加密后的拷贝代码，这样就能够混淆杀毒软件

变形代码 metamorphic code：每次传播时都会改变自己的形式

防御方法：

  • 行为检测
  • 标记不熟悉的代码

反病毒公司有时为了夸大自己的重要性，可能会修改统计标准，把一个病毒的多种变形统计为多种病毒

Tor

Internet 上的 匿名性 anonymity 是非常困难的

实现匿名的基本思想：让别人帮你做事

Tor The Onion Router：一个匿名网络，使用多重代理（中继）来实现匿名通信

这样，通信过程中的每一个中继都只知道前一个中继和后一个中继，而不知道通信的双方

Tor 的缺点：能够对整个网络有全局掌握的攻击者可以得知通信的双方，每当 A 发出某个信息后，B 收到了一条信息，所以仍然可以推测此时 A 和 B 正在通信（边信道攻击）

事实上，Tor 的威胁模型只考虑了防御有局部视角的攻击者，而没有考虑到全局视角的攻击者

另一个问题在于 collusion，即多个中继合作来破坏匿名性

此外，Tor 无法隐藏你正在使用 Tor 的事实

Tor Onion Service：一个隐藏服务，用户可以通过 Tor 访问，但是服务端不知道用户的 IP 地址
