Clean Code

发表于2024-12-19|更新于2024-12-19|工具
|总字数:876|阅读时长:2分钟

基于 Clean Code 编写

有意义的名字

使用揭示意图的名字

避免虚假信息

做有意义的区分

使用能发音的名字（方便交流）

方便搜索

避免编码（多余的前缀或后缀）

每个单词只用在一个概念上

一个概念只使用一个名字

使用解决域的名字便于快速理解意图

但通常没有现成的解决域，不要新造术语，改为使用问题域的名字

名字的出现一定伴随着上下文，要添加有意义的上下文

函数

小，通常是让程序员一页能放得下

只做一件事

每个函数中调用的过程都处于同一层抽象上

描述性名字

函数参数尽量不要多，越少越好

将有副作用和没有副作用的函数分开，做出明显的标识

尽量避免输出参数

查询和命令分离

倾向异常而不是返回错误码

  • 提取 try/catch 块，因为错误处理已经是一件事了

不要重复自己

结构化编程

注释

注释无法装扮坏代码，应该在代码中解释

好的注释：

  • 法律注释
  • 告知性注释
  • 解释意图
  • 澄清
  • 警告后果
  • TODO
  • 放大容易被忽略的重要点
  • 公共 API 的 doc

坏的注释：

  • 说的不清楚
  • 多余
  • 假注释
  • 强制性注释
  • 日志
  • 噪音
  • 在可以使用函数或变量时，不要使用注释
  • 注释掉的代码
  • 非局部信息
  • 太多信息
  • 不明显的联系

格式化

垂直格式化：

  • 概念之间的垂直上要有空行开放
  • 垂直上的信息密度
  • 变量、依赖的函数在垂直上的声明-使用距离
  • 概念密切

水平格式化：

  • 水平上的开放和密度
  • 缩进
  • 对齐

对象和数据结构

数据和对象是反对称的，即：

  • 过程式代码便于在不改变原有的数据结构的前提下，添加新的函数，面向对象便于在不改变现有的函数的前提下，添加新的类
  • 相反，过程式代码难以添加新的数据结构，因为所有函数都必须改变；OO 代码难以添加新的函数，因为所有的类都会改变

通常情况下，添加新的数据类型更多，因此 OO 在很长一段时间占据主流地位

一个模块不应该了解操作对象的内部结构

DTO 只有公共变量，没有函数

错误处理

使用异常

先写 try-catch-finally

提供异常的上下文

定义异常类

定义正常的流

不要返回 Null 和传递 Null

单元测试

TDD 三大法则：

  1. 在编写失败的单元测试之前，不要写产品的代码
  2. 不要写多于足够让代码失败的单元测试
  3. 不要写多于足够通过当前失败的测试的产品代码

测试代码本身也要保持 clean

双重标准

每个测试一条 Assert

FIRST 规则：

  • 快速
  • 独立
  • 可重复
  • 自我验证
  • 及时

类

类组织：封装

类同样应该小：

  • 单一职责
  • 内聚

为了改变而组织

  • 从改变中孤立出来

系统

将构建系统从使用中分离开

  • 将 main 分离
  • 工厂
  • 依赖注入

系统需要领域特定语言 DSL 来充当桥梁
